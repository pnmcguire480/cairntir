## Prehistory — the two predecessors

### BrainStormer (`c:/Dev/SKILLS/BrainStormer/`)

Patrick's prior attempt at a thinking layer for Claude. It contributed Cairntir's **vocabulary**: Crucible (epistemic forge), Quality (PALADIN ship gate), agent species, ETHOS. It also contributed the negative lessons that shaped the rebuild: 224 silent `except: pass` blocks, dead license-validation code paths, and the fundamental mismatch the project itself eventually named "the architecture of a learning system but the runtime of a static scaffolder."

What Cairntir kept: the three skill names, the philosophy that comprehension precedes code, the ETHOS principles.
What Cairntir refused: the runtime. Not one line of BrainStormer Python is imported. The skills were re-distilled from the markdown, not ported from the implementation. `lineage/brainstormer/` exists read-only as historical reference.

### MemPalace (https://github.com/milla-jovovich/mempalace)

A separate project Patrick studied. MemPalace shipped a brilliant **memory taxonomy** (wings → rooms → drawers, four-layer retrieval: identity / essential / on-demand / deep) and a 96.6% recall@5 score on LongMemEval. What it did not have: a reasoning layer. It is a beautifully organized store, not an active participant.

What Cairntir kept: the taxonomy, the four-layer retrieval, the `sqlite-vec` backing decision (verified independently). The concepts were studied, not the code.
What Cairntir refused to import: the codebase itself. `lineage/mempalace.md` documents what we learned and where we diverged.

> **The differentiator, in one line:** MemPalace remembers. Cairntir
> remembers *and reasons*. The reasoning layer (three skills,
> prediction-bound drawers, executable recipes) is the gift Cairntir
> exists to give.

## 2026-04-08 — the round table

Eight thinkers reviewed the original "Long Road" plan in one session: Karpathy,
LeCun, Sutskever, Hinton, Fuller, Peter Joseph, Alan Watts, Uncle Bob.
They converged on five themes that became the **committed v0.2 → v1.0 arc**:

1. **Prediction-bound drawers** — every drawer can carry `claim`,
   `predicted_outcome`, `observed_outcome`, `delta`, `supersedes_id`. The
   AutoResearch loop's substrate. Karpathy / LeCun / Hinton / Sutskever /
   Fuller / Uncle Bob all flagged this independently.
2. **Consolidation + forgetting** — verbatim is the floor, not the ceiling.
   Hinton / Watts / LeCun.
3. **Surprise as the load-bearing field** — `delta` is the gradient when
   there are no weights. LeCun / Fuller / Hinton / Karpathy.
4. **Portable signed format = anti-capture** — format is the product, not
   the implementation. Peter Joseph / Fuller.
5. **Cut Team Memory as a feature** — replicable beats shared. Falls out
   of the portable format for free. Karpathy / Fuller.

Decision rule that stuck: when 8 thinkers converge, the convergence is
committed, not advisory. `feedback_round_table_consensus.md` in
auto-memory captures this rule for every future session.

## v0.2 — Prediction-Bound Drawers + Eval-on-PR

Schema migration adds five optional fields (`claim`, `predicted_outcome`,
`observed_outcome`, `delta`, `supersedes_id`). `DrawerStore` gains a
`SCHEMA_VERSION = 2` constant and forward-only `_migrate()` pass.
Reason skill gains a mandatory Step 4.5 ("predict") — no decision leaves
the loop without a falsifiable claim.

LongMemEval R@5 wired into CI as a separate `eval` job that fails on
regression.

## v0.3 — Consolidation, Forgetting, Contradiction

Schema v3 adds `last_accessed_at` and `access_count`, backfilled from
`created_at`/`0` for pre-v3 rows. New module `cairntir.memory.consolidate`
exposes three pure functions: `demote_stale` (drift cold drawers down a
layer), `detect_contradictions` (flag, never average), `consolidate_room`
(emit derived ESSENTIAL drawer with citations).

## v0.4 — Surprise & Belief-as-Distribution

Schema v4 adds `belief_mass: float = 1.0` (clamped at zero). Two new
store methods (`reinforce` / `weaken`). New `cairntir.memory.belief`
module folds raw vector distance through `mass * (1 + delta_boost)`.
`store.search()` reranks by belief by default. The math is deliberately
blunt — belief is a steering wheel, not an engine.

## v0.5 — Portable Signed Format (Anti-Capture Lock)

`cairntir.portable` module: versioned envelope with sha256 content
addressing and optional HMAC-SHA256 signatures. Structural prohibition
enforced by `ensure_no_external_urls()` — only `cairntir://` references
are allowed. Export fails closed on a single violating drawer.

`cairntir export` and `cairntir import` over any substrate (USB, IPFS,
git, mailing list). The format is the product.

## v0.6 — Reason Loop Through Clean Ports

`cairntir.reason` package: four runtime-checkable Protocol ports
(`HypothesisProposer`, `ExperimentRunner`, `BeliefStore`,
`MemoryGateway`) and a pure `ReasonLoop.step()` that orchestrates one
predict→observe→update cycle without importing sqlite, networks, or
LLMs. Production wiring lives outside the library.

The loop writes two drawers per step, always, even when the runner
fails. **Verbatim is the floor; a failed step is not a skipped step.**

## v1.0.0 (2026-04-08, public on GitHub 2026-04-14)

Library extraction. Cairntir graduates from "a tool" to "the thing
other tools store their memory in."

- `cairntir.__init__` exposes ONLY the stable seam: protocols
  (`Store`, `EmbeddingProvider`, the four reason-loop ports), frozen
  value types, typed exceptions, `__version__`. Concrete impls move
  to `cairntir.impl.*`, reserved-right-to-change.
- New `CairntirDeprecationWarning` subclass of `DeprecationWarning`.
  Public surfaces must emit it for two minor releases before removal.
- Contract test suite, property tests via Hypothesis, public-API
  snapshot test that fails on `dir(cairntir)` drift.
- Migration fixtures kept in tree forever, one per prior schema
  version.

138 tests passing, 88% coverage, ruff + mypy --strict + silent-except
scanner all clean. Tagged, GitHub release cut.

## v1.0.1 (2026-04-17) — install hardening

The first sharp edge surfaced after PyPI publish: opening a fresh
Claude Code session in `c:/Dev/Cairntir/` showed *"Cairntir MCP tools
are not loaded in this session — you may want to run `cairntir init
--user --force`."*

The user's framing was decisive: **"cairntir TRUE until cairntir
FALSE, user controlled."** No new tools. Fix permanently.

Root cause: 1.0.0's MCP registration pinned to `sys.executable`,
which silently breaks whenever the venv moves, gets recreated, or the
Python version upgrades. Claude Code spawns the registered command;
the dead path looks like "no tools" with no surfaced error.

**Three fixes shipped in one commit (`9fdf678`):**

1. **`cairntir-mcp` console script** declared in
   `[project.scripts]`. Pip's launcher hard-pins the right
   interpreter on install. The registration now points at this
   stable name.
2. **`cairntir.register` module** — silent self-heal called from the
   CLI root callback on every invocation. Checks `claude mcp list`,
   re-registers if missing, writes a checkpoint at
   `cairntir_home() / .registered` for fast-path no-ops. Opt-out via
   `CAIRNTIR_DISABLE_AUTOREGISTER`.
3. **`cairntir.update` module** — non-blocking PyPI update notifier.
   Daemon-thread check (24h cache, 2s timeout, fail-silent). When a
   newer release exists, the next CLI command and the next MCP tool
   response prepend a one-line banner. Opt-out via
   `CAIRNTIR_DISABLE_UPDATE_CHECK`.

Plus: dropped the inaccurate "Phase-2 stub" labels on `cairntir_audit`
and `cairntir_crucible`. Plus: an autouse conftest fixture so the
silent side effects never touch the dev machine or hit the network
during test runs.

188 tests passing (was 166), 88% coverage, all gates clean.
Independent Opus 4.7 post-fix audit: zero ship-blocking issues.

PR #11 merged via rebase to keep history linear.

## The cold-start fix (post-1.0.1, on `main` as `d505e99`)

After 1.0.1 landed and the user reinstalled, the tools *still* did
not load on the next Claude Code session. Empirical diagnosis from a
prior session (handed off as a lesson): **MCP servers must answer
the JSON-RPC `initialize` handshake within ~10 seconds. Heavy
imports at module level eat the budget.**

`DrawerStore.__init__` had been eagerly calling
`embedder.dimension`, which on `SentenceTransformerProvider` cold-loads
torch + sentence-transformers + BertModel. Measured at **28 seconds**.
Claude Code's connection timeout fires at ~10s. The server appeared
crashed and all six tools failed to register.

Fix: defer the embedder load until it is genuinely required.

- `DrawerStore.__init__` sets `self._dim = None` and stops touching
  the embedder. Docstring documents the 10s/25s tradeoff so the next
  refactorer knows what they would break.
- `DrawerStore._init_schema` reads `embedder.dimension` *only* when
  the `vec_drawers` virtual table must be created (first-time DB).
  Existing DBs — the common path — never load the model at startup.
- `DrawerStore.add` lazily populates `self._dim` from the embedder
  *after* `embed()` has already warmed the model, so the read is free.

Empirical: 28s startup → 1s startup. All six tools register cleanly.
PR #12 merged via rebase. Local `main` now at `d505e99`.

The lesson is preserved as a permanent feedback memory at
`auto-memory/feedback_mcp_cold_start.md`. Every future MCP-adjacent
build in any wing should consult it.

## v1.1 — the synergy stack (planned 2026-04-17)

The user's question: *"what on the roadmap would get us the biggest
wins? i want to pull the 3 most influencial upgrades forward."*

Answer: pull v1.2 (Production Reason Loop) + v1.3-partial (Cross-Wing
Recall + Temporal) + v1.5 (Recipe Runtime) forward and ship them
**together**, because the value compounds when they land in the same
release.

- A recipe needs the production reason loop to actually reason.
- A reason loop is more demoable as the executor of a recipe than as
  a bare CLI command.
- Both are more powerful when their memory layer can pull from every
  wing the user has ever written to.

Plan committed to `plans/v1.1-synergy-stack.md`. Implementation order:
#2 (cross-wing recall, smallest, no new deps) → #1 (production reason
loop, adds optional `anthropic` dep) → #3 (recipe runtime, depends on
both). Bundled with the cold-start fix as the first 1.1.0 release —
PyPI is still at 1.0.0, so 1.1.0 becomes the first published release
that has both halves of the install hardening *and* the synergy stack.

The user's framing of why this matters: *"this is what differentiates
us from MemPalace. that's what i wanted to give to others."*

## What this document is for

If you are an AI agent or human reading this on a fresh chat:

- The `Drawer` schema, the four-layer retrieval, the `sqlite-vec`
  backing — those are MemPalace's contribution, studied and
  reimplemented.
- The three skills, the predict-bound drawer rule, the verbatim-floor
  ethos — those are BrainStormer's contribution, distilled and
  rebuilt without inheriting its runtime mistakes.
- The install hardening (`cairntir-mcp` shim, silent self-heal),
  the cold-start fix (lazy embedder), the v1.1 synergy stack — those
  are decisions made on 2026-04-17 that you should not relitigate
  unless something underneath them broke.
- The version comparator's pre-release nudging behavior, the silent
  fail of network/disk operations in the update notifier and
  self-heal — those are documented trade-offs, not bugs. Read the
  module docstrings before "fixing" them.

If something contradicts what is written here, trust the code first
(it might have moved on) and update this document when you do.
