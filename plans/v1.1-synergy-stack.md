# Plan — v1.1 Synergy Stack

> Status: **active**. Approved 2026-04-17. Pulls v1.2 / v1.3-partial / v1.5
> from the Road to 2.0 forward and ships them together because the value
> compounds when they land in the same release.

## Why these three, why now

The roadmap committed v1.2 (Production Reason Loop), v1.3 (Cross-Wing +
Temporal), and v1.5 (Recipe Runtime) as separate phases. Shipped solo
each is useful. Shipped together they turn Cairntir from a passive
per-project store into an active cross-project reasoning system:

- **Production Reason Loop** gives Cairntir a thinking personality.
  v0.6 shipped clean ports and zero concrete adapters; the loop cannot
  actually reason today.
- **Cross-Wing Recall + Temporal** makes the memory globally
  accessible. A solution found in `stars-2026` becomes recallable from
  `ground-zero` when the same shape comes up.
- **Recipe Runtime** makes the protocols executable instead of
  manually walked. Signal Reader becomes one command, not seven steps.

The stack is the multiplier:

- A recipe needs the production reason loop to actually reason.
- A reason loop is more demoable as the executor of a recipe than as
  a bare CLI command.
- Both are more powerful when their memory layer can pull from every
  wing the user has ever written to.

## Scope (a tight v1.1.0)

Bundles the three upgrades plus the cold-start fix already on `main`
(d505e99) into one release. Skips 1.0.2 — semver says new features
warrant a minor bump, and 1.0.1 has not been published to PyPI anyway,
so 1.1.0 becomes the first published release that has both halves of
the install hardening *and* the synergy stack.

## Implementation order (smallest first, each unlocks the next)

### #2 — Cross-Wing Recall + Temporal Walk *(first, no new deps)*

- `DrawerStore.search()` already accepts `wing=None`. Expose it
  through MCP as a new tool: `cairntir_cross_recall(query, limit)`.
  Returns hits annotated with wing-of-origin so the caller can
  reason about provenance.
- `cairntir.memory.temporal` module: `walk_supersedes(store, drawer_id)`
  returns the chain root → leaves; `as_of(store, drawer_id, when)`
  returns the drawer that was the leaf at timestamp `when`. No schema
  change — pure query layer over `supersedes_id`.
- New backend method `CairntirBackend.cross_recall` mirrors the
  existing `recall` but searches every wing.
- Tests: cross-wing search returns hits from both wings; temporal walk
  follows multi-step supersedes chains; as-of returns the correct
  predecessor for a timestamp between two observations.

### #1 — Production Reason Loop *(second, adds optional `anthropic` dep)*

- New sub-package `cairntir.production` with three concrete adapters:
  - `ClaudeProposer(api_key, model="claude-opus-4-7")` — implements
    `HypothesisProposer` by calling the Anthropic Messages API with a
    pinned system prompt. Returns a `Hypothesis` with a non-empty
    `predicted_outcome` (validates and re-prompts once if empty).
  - `StoreBackedMemory(store)` — implements `MemoryGateway` by
    delegating to a `DrawerStore`. Trivial wrapper.
  - `StoreBackedBeliefs(store)` — implements `BeliefStore` by calling
    `store.reinforce` / `store.weaken`. Already shape-compatible.
- A `NullRunner` adapter (zero-cost `ExperimentRunner`) that records
  the user's manual verdict — useful for recipes where the
  "experiment" is a human reading the output.
- New CLI command: `cairntir reason "<question>" --wing X [--room Y]`
  that wires `ClaudeProposer + NullRunner + StoreBackedBeliefs +
  StoreBackedMemory` and runs one full step.
- `anthropic>=0.40` becomes a `[project.optional-dependencies]` entry
  under a new `production` extra. Importing `cairntir.production`
  without it raises a clean `ImportError` pointing at
  `pip install cairntir[production]`.
- Tests: ClaudeProposer with a fake HTTP transport (no real network
  call) verifies the prompt shape and the empty-prediction retry;
  StoreBackedMemory + Beliefs are integration-tested against a real
  `DrawerStore` over a tmp DB; the `cairntir reason` command runs
  end-to-end against fakes.

### #3 — Recipe Runtime *(last, depends on both)*

- `cairntir.recipes` module:
  - `RecipeContract` dataclass loaded from `recipe.toml`: `name`,
    `description`, `input_schema`, `output_wing`, `skills` (list of
    `crucible | quality | reason`), `version`.
  - `RecipeRunner(store, proposer, runner)` — given a contract and an
    input dict, executes the chain and returns a `RecipeResult` with
    every drawer id written.
- New `recipe.toml` for `signal-reader` describing the five-step
  protocol declaratively (input = url + summary, chains crucible +
  reason, output_wing = signals).
- New CLI commands:
  - `cairntir recipe list` — finds every `recipe.toml` under
    `docs/recipes/` plus user-level `~/.claude/recipes/`.
  - `cairntir recipe run <name> --input KEY=VALUE [...]` — validates
    input against the contract's schema, executes, prints a verdict
    plus drawer ids.
- Tests: contract loader rejects malformed TOML, missing fields,
  unknown skills; runner executes a fake recipe end-to-end against
  fake adapters; `cairntir recipe list` discovers the bundled
  signal-reader recipe.

## Stuff not in this plan

- **SandboxRunner** (a real `ExperimentRunner` that executes shell
  commands) is deferred — `NullRunner` is enough to ship the recipe
  runtime, and a sandbox is its own security project.
- **Calibration dashboard** (`cairntir calibration --wing X`) is
  deferred — useful but not load-bearing for the synergy story; can
  ship in v1.2 once enough prediction-bound drawers exist to be
  worth visualizing.
- **Cross-wing belief reranking** — `cross_recall` returns raw
  semantic distance for now; per-wing belief mass cross-comparison
  is a research question, not a v1.1 commitment.
- **Recipe input validation past basic types** — JSON Schema-style
  validation can land later if recipes get complex.

## Acceptance criteria

- `pytest -q` green, coverage ≥ 80%
- `ruff check src tests` clean
- `mypy --strict src` clean
- `scripts/check_no_silent_except.py` clean
- `cairntir cross-recall "<query>"` and `cairntir reason "<q>" --wing X`
  and `cairntir recipe run signal-reader --input url=...` all execute
  end-to-end against real stores in a smoke script
- CHANGELOG entry under `[1.1.0]`
- Roadmap updated: v1.2 / v1.3-partial / v1.5 marked as **delivered
  ahead of schedule in v1.1**, leaving v1.4 (file sync) + v1.6 (agent
  memory) as the path to v2.0
- One PR for the whole stack, three commits inside it (one per
  upgrade) so the bisect story stays clean

## Out-of-scope guardrails (so the next session does not re-litigate)

- **Three skills stays locked.** Recipes are the escape valve. If a
  fourth core primitive is tempting, write a recipe instead.
- **No SaaS, no hosted service.** Anthropic API calls happen against
  the user's own key, on their own machine, no proxy.
- **No telemetry from the production loop.** What the user reasons
  about is theirs.
- **Verbatim is the floor.** Production adapters add machinery on top
  of the verbatim store; they never paraphrase what the proposer
  returned.
